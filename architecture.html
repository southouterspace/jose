Two machines, one seam.

The parametric engine and the drawing engine are not one program fighting itself. They are two machines on opposite sides of a thread boundary — and the stack works because they stay apart.

The pipeline you already designed is the parametric engine. Each stage is dirty-driven off the one before it. Nothing new to invent — just make that chain live, incremental, and resident in Rust. Only the first stage (capturing the edit) and the last (laying out the sheet) touch the main thread.

The split

One owns truth. One owns feel. The dashed spine between them is the only place they touch.

The boundary contract

Two channels, never one. The mistake is running the model through RPC. Calls are cheap and occasional; the model is huge and constant. So they don't travel the same road.

Channel A — calls

Commands & events via wasm-bindgen

Low volume — one per user action. JS sends intents ("push face N by 3½″"); Rust returns which buffer ranges changed. Ergonomics and compile-time type safety win here; the copy cost is negligible at this volume.

JS → intent → Rust · Rust → dirty-ranges → JS
Channel B — state

The model via shared linear memory

Rust owns the SoA tick arrays in WASM memory; JS makes typed-array views over the same bytes to render. Zero-copy, no marshaling, every frame. Layout offsets are generated from the one MODEL source so the two sides can't drift.

one buffer · read-only from JS · pass by reference, never by copy

Why dragging never feels the boundary

The fear: every drag frame round-trips to Rust and the recompute makes it lag. The fix: preview never crosses the seam. Two phases.

JS · main thread
Drag begins — the ghost moves
The drawing engine moves geometry visually only. Cheap, throwaway, instant. Rust is not consulted. This is the SketchUp trick: drag feedback is a lightweight overlay, not a real solve.
JS · main thread
Snapping resolves locally
Per-mouse-move snap queries hit a local spatial index over the render mirror. Pixel-defined tolerance. Never round-trips.
Seam · on release / typed value
Intent crosses as one command
Now — and only now — the committed op travels Channel A. One message, not one-per-frame.
Rust · worker
Dirty propagation, incremental
The dependency graph walks from the changed node and resolves only what depends on it — reframe studs, resize header, re-roll loads, re-check, re-optimize cuts. Coalesced recompute at model scale.
Rust · worker
Writes results, returns dirty ranges
Updated geometry lands in shared memory. Rust hands back only the changed buffer spans.
JS · main thread
Ghost swaps for truth, dirty-only redraw
The drawing engine drops the overlay and renders the real result — only the ranges that changed. The new framing appears here, on commit, not during the drag.

Critical placement review

The naive line is "interaction → JS, compute → Rust." Mostly right. But five things don't sit where that rule first puts them. Tap to expand.

01
The snap acceleration structure
Naive call: "snapping is interaction, so it's all JS."
Split
▸
Build
Rust, on commit — the spatial index (BVH) over thousands of members is real geometry work.
Query
JS, per mouse-move — reads the structure straight out of shared memory.
Why
If the index lived only in Rust, every mousemove would round-trip and snapping would lag. If it were built in JS, you'd rebuild a heavy structure on the render thread. Build heavy in Rust, query light in JS. The structure is shared; the lookup is local.
02
Undo / redo history
Naive call: "commands originate in JS, so JS keeps the log."
Rust
▸
Owner
Rust — whoever owns canonical state owns undo.
JS role
Sends an "undo" intent. That's all.
Why
Undo must restore the model exactly, including every derived recompute (loads, cuts, checks). A JS-side command log couldn't reconstruct that without replaying through Rust anyway — so the authoritative inverse log lives where the state lives. Transient in-progress tool state stays in JS; committed history is Rust's.
03
Document export — the "one true 2D path"
Naive call: "2D output, so it's a JS concern."
Split
▸
Rust
Projection + hidden-line removal. HLR over a BREP solid is heavy geometry — it belongs next to the kernel.
JS
Sheet composition — line weights, dimensions, hatching, title block, SVG/PDF assembly.
Why
"2D" hid two different jobs. The math of what lines are visible from this view is geometry (Rust). The craft of arranging them on a D-size sheet is layout (JS). Non-interactive and derived either way — runs on the one-way chain, not per frame.
04
Preview geometry during a drag
Naive call: "show the user the real result as they drag."
JS — and deliberately dumb
▸
Does
Moves existing geometry visually. Instant.
Does not
Reframe. New studs, resized headers, re-rolled loads do not appear mid-drag.
Why
An accurate live preview would need the full solver every frame — the exact lag we're avoiding. So the ghost is honest about being a ghost: it shows the move, not the consequences. Consequences land on release. This is a UX decision, stated on purpose, not a limitation to apologize for.
05
Structural check results
Naive call: "checking is compute (Rust), done."
Split
▸
Compute
Rust — NDS math produces a pass/fail flag per member, written into shared memory.
Surface
JS — reads the flag and colors the member (overstressed → red), draws the callout.
Why
The verdict is data; the way you see the verdict is rendering. Keeping the flag in the shared buffer means the highlight costs nothing extra — it's just another column the render mirror already reads.

The two rules that keep them from fighting

Rust owns canonical geometry. JS owns a mirror.
One direction: parametric → render. JS never mutates the model — it emits intents and reads results. Single source of truth holds across the boundary.
Anything per-mouse-move stays local.
Snap queries, preview, picking — all run on the main thread against shared memory. The seam is crossed on commit, never on motion. That's what makes one tool out of two machines.
